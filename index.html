Section 01 · Introduction
Next: motivation
Core argument: automation codifies known solutions to known problems. The hard work in HPC is *novel* combinations of symptoms that scripts have never seen. This is where an LLM-based reasoner genuinely adds value — synthesis across layers, not just rote command execution.
Introduction · Motivation

Why intelligence, not just automation?

  • Scripts solve known problems on known inputs — they break on novelty.
  • Runbooks encode expert knowledge but don't adapt to new symptoms.
  • HPC issues are usually cross-layer: scheduler + network + filesystem + application.
  • Operators need systems that reason about context, pick the right tool, and iterate.
  • LLM-based agents synthesize logs, metrics, and docs the way a senior engineer does.
  • The goal isn't replacement — it's amplification of expert time.
Rule of thumb: if the failure pattern fits on a wiki page, automate it. If it requires a senior engineer to think, an agent may earn its place.

SECTION 02What are AI Agents?

An agent is more than a chatbot.

An AI agent is a system where an LLM decides what to do next — choosing tools, reading results, updating its plan, and iterating toward a goal.

LLM Reasoning

The brain — interprets intent, plans, evaluates evidence, revises.

Tools

APIs, CLIs, schedulers — how the agent acts on the world.

Memory

Short-term context + long-term vector store of incidents & docs.

Control Loop

Observe → Think → Plan → Act → Read → Respond — looping until the goal, budget, or human check.

Agent = LLM + Tools + Memory + Control Loop.  Everything else — frameworks, protocols, orchestration — is scaffolding around this core.

What are AI Agents? · Taxonomy

Types of agents.

Reactive

Responds to a single trigger. Bounded, short-lived, highly observable. "Answer this ticket using runbook + logs."

Low risk · High ROI

Autonomous

Pursues a standing goal over time. "Keep the scratch filesystem under 85% used." Observes, decides, acts, learns.

Medium risk · Needs SLOs

Multi-agent

Specialized agents coordinate: scheduler, storage, network, orchestrator. Richer behavior, harder to debug.

High risk · High ceiling
Maturity path: start reactive → graduate to autonomous with strict SLOs → only then consider multi-agent with explicit handoffs.

Section 03 · Architecture
Next: the reasoning loop
ReAct-style loop. Explain: at each iteration, the LLM emits a *thought*, then either a *tool_call* or a *final answer*. The environment returns a result, which becomes the next observation. The loop terminates when the model decides it has enough — or when we hit a budget.
How Agents Work · ReAct

The reasoning loop.

At each step the LLM asks: do I have enough information? If not, which tool next?

agent · trace
reasoning
// iteration 1 · full pass through the loop
① observe: user asks "why is gpu job 842193 pending?" (18m wait)
② think: candidates → queue policy, host state, per-user quota
③ plan: bjobs → bhosts (A100_80GB) → bqueues → decide
④ act: run_cmd("bjobs -l 842193")
⑤ read: PEND · gmodel=A100_80GB×4 · "not enough hosts w/ requested GPU"
// iteration 2 · loop back — more evidence needed
② think: fleet shortage, or just our timing?
④ act: run_cmd("bhosts -gpu | grep A100_80GB")
⑤ read: 1 host closed_LIM, 1 host w/ 3 free — capacity tight, not zero
// iteration 3 · capacity is fine, look at policy
② think: must be a user- or queue-level cap
④ act: run_cmd("bqueues -l gpu-priority")
⑤ read: MAX_JOBS/user=2 and alice RUNNING=2 — that's the wall
⑥ respond: root cause = user MAX_JOBS; 3 options drafted (wait/resize/switch)
Section 03 · ReAct Loop
Next: memory
Memory is often the difference between a toy and a useful agent. Short-term memory lives in the context window and is bounded — requires summarization. Long-term memory is a vector DB of past incidents, docs, topology. The payoff: an agent that says "we saw this on gpu-node-147 two weeks ago."
How Agents Work · Memory

Memory: short-term & long-term.

Short-term (working)

The current conversation and tool outputs. Lives in the LLM context window. Bounded by tokens — survives via summarization, structured scratchpads, and careful eviction.

window · ~100–200k tokens

Long-term (persistent)

Vector store of past incidents, scheduler docs, cluster topology, runbooks. Retrieved via semantic search so the agent can say "we saw this ECC signature last Tuesday on gpu-node-147."

vector · RAG · graph
Key insight: an agent without memory is a stateless oracle. An agent with memory becomes an institutional knowledge amplifier — and the amplifier gets stronger with every incident.
Section 03 · Memory
Next: planning
Brief tour of planning strategies. Cover ReAct, plan-then-execute, reflection, tool schemas, budgeting, HIL. Emphasize: budgeting prevents runaway; HIL keeps humans in charge for anything destructive.
How Agents Work · Planning

Planning & iteration.

  • ReAct — interleaves thought and action, one small step at a time.
  • Plan-then-execute — draft a full plan, run it, reflect on failures.
  • Reflection — critique each outcome and revise the approach.
  • Tool schemas (JSON) make tool use reliable and machine-checkable.
  • Budgets — cap steps, tool calls, tokens, and wall-clock to bound cost & blast radius.
  • Human-in-the-loop — destructive actions always require explicit approval.
A good agent is boring on the happy path and loud on the unusual one. Design for the unusual one first.

Agents in HPC · Tool Surface

The HPC tool surface.

● Read-only (default)

  • bjobs, bhosts, bqueues, lsload, lshosts, bhist
  • sinfo, squeue, sacct, scontrol show
  • mmlsdisk, mmhealth, mmdiag (GPFS)
  • nvidia-smi, dcgmi, ibstat, perfquery
  • Prometheus / Grafana / Loki APIs, ServiceNow / Jira read
  • Docs / wiki retrieval via embeddings

● Write (gated + approved)

  • bmod, bkill, brequeue — with explicit user approval
  • badmin, scontrol update — operator-only, dry-run first
  • Open / update ticket with an evidence bundle
  • Post to on-call channel with structured summary
  • Trigger a runbook step — only via an allow-list
  • Every write call: logged, reviewable, reversible where possible
Rule of thumb: the agent observes freely but acts only with explicit scope, approval, and audit. Dry-run is a first-class mode, not an afterthought.

Section 06 · Reference setup
Next: the flywheel
This is the "invest once, compound forever" slide — the argument for why the agent gets strictly more valuable over time instead of going stale. Three beats. (1) SELF-IMPROVING. Every resolved ticket is a training example we already paid for. The loop: user query → agent reasons → tools → proposed answer → operator accepts / corrects / closes → the *outcome* (symptom signature, tools actually used, root cause, operator correction) is written back to long-term memory as a reusable "case". Next time a similar symptom appears, the retriever surfaces that case in the very first hop — the agent doesn't re-derive, it cites. This is the same pattern as Nous Research's Hermes agent ("the agent that grows with you" — autonomously distils skills from experience and persists them across sessions) and the Reflexion framework (Shinn et al., 2023) where verbal self-reflection over past trajectories is stored in an episodic buffer and retrieved on future attempts. We are not inventing anything exotic — we are writing back what every operator already writes in their own head to the ticket's "resolution" field. (2) MODEL-SWAPPABLE. The only contract the agent has with the brain is the OpenAI-compatible /v1/chat/completions HTTP endpoint. The model behind it is an implementation detail. Qwen3.6-35B-A3B today because it's the best open-weights fit for our 2× A100 40GB. When Qwen4 / Llama5 / something-else-entirely ships in six months and wins our internal eval, we rebuild the vLLM container with the new weights, re-run the regression replay against our 200 canonical incidents, and cut over. Zero changes to the agent, tools, memory, or anything a user sees. During rollout we can even serve both models on a split and gradient — the infrastructure is the investment, the weights are rental. (3) TWO AGENTS, ONE BRAIN. Same stack serves two very different audiences with different safety envelopes. END USERS (researchers, grad students, scientists) get a deliberately bounded agent: narrow tool surface (bjobs / bhosts / queue status / docs search), read-only by default, tight token / turn budgets, safe defaults, clear refusal messages. It's the assistant that answers "why is my job pending?" and "how do I request an A100?" without ever being one flag away from a catastrophic write. ADMINS AND OPERATORS get a materially more powerful agent on the same infrastructure: full CLI surface, replay and diagnostic tools, write actions (bmod, config changes, runbook steps) behind explicit approval gates with dry-run previews, larger budgets so it can chase deeper multi-hop investigations. CRUCIALLY — both agents share the same institutional memory. When an admin resolves a gnarly NVLink fabric issue on Tuesday, the user agent can cite the relevant resolution (appropriately redacted for policy and hostnames) when a user hits the same symptom on Wednesday. The more the ops team uses it, the smarter the user-facing surface gets, automatically. Close with the one-liner: "the more Tuesdays you run it, the more Wednesdays pay off."

SECTION 07Summary

Key takeaways.

  • An agent = LLM + Tools + Memory + Control Loop.
  • HPC is a natural fit: rich telemetry, well-defined CLIs, high operator cost, knowledge-heavy.
  • Start reactive and read-only; add write capabilities behind explicit approval gates.
  • Memory of past incidents is what turns a toy into an institutional assistant.
  • Design for auditability, budgets, and human-in-the-loop from day one.
  • Measure success in operator time saved and MTTR, not tokens burned.
  • Host it on the GPUs we already own — 2× A100 40GB + Qwen3.6-35B-A3B (Apache 2.0), zero per-token bill, data never leaves the building.
  • It compounds: every resolved ticket sharpens the next one, the brain is swappable as better open-weights ship, and one shared memory serves both a bounded user agent and a powerful admin agent.

Summary · Limitations

What agents are not good at (yet).

Hallucinations

LLMs invent commands, flags, and paths. Ground with retrieval and validate every tool call schema.

Safety & blast radius

One wrong bkill -u all is catastrophic. Default read-only; gate writes with dry-run + approval.

Control & oversight

Audit logs, reproducibility, deterministic fallbacks — mandatory, not optional.

Cost & latency

Token and time budgets matter. Not every question deserves a 20-step agent.

Long-horizon drift

Agents drift on multi-hour plans. Break into smaller contracts with clear exit criteria.

Evaluation

"Was the agent useful?" is hard to score. Invest in golden scenarios and regression replay.

Sidebar · LLM Basics

What is an LLM?

A neural network that has read most of the internet — and learned, one token at a time, to predict what comes next. Everything else an “AI” seems to do emerges from that one trick.

Parameters

35 billion weights

Numbers tuned during training. Collectively they encode grammar, facts, code patterns, reasoning habits — compressed from trillions of words of text.

Training data

~36 trillion tokens

Books, code, the open web, scientific papers — the full Qwen3-series pre-training corpus. The model never memorizes the text verbatim; it learns the shape of language and knowledge.

Context window

262,144 tokens

How much it can read in one go. Roughly a 500-page book. Big enough to hold a full ticket thread plus cluster docs and still reason across them.

Prompt → Next-token probability

Why is my GPU job pending ?
It isn't looking up answers in a database. It's running a function — tokens → probabilities — and drawing one sample at a time. Everything else (reasoning, tool use, code) is scaffolding built on top of that single primitive.

How they learn.

Three stages turn a blank neural network into a helpful assistant. Open-weight models like Qwen3.6 arrive with all three already done — you just download them.

1
PRE-TRAINING · months · thousands of GPUs

Predict the next token

Shown tens of trillions of tokens of text (~36T for Qwen3-class) with one job: guess what's next. From this emerges grammar, facts, code, reasoning, common sense. Costs tens of millions of dollars. Done once by the model lab.

2
FINE-TUNING · days · curated data

Learn to follow instructions

Trained on “instruction → ideal answer” pairs. Teaches the model to answer questions instead of just continuing them, to use a consistent format, to call tools, to refuse unsafe asks.

3
PREFERENCE TUNING · days · human feedback

Align with human taste

Humans rank pairs of answers; the model learns which style people prefer (RLHF, DPO). This is the difference between a model that's technically right and one that actually feels useful.

Key insight: only stage 1 is expensive. Stages 2–3 are cheap. That's why the open-weight ecosystem moves so fast — labs share pre-trained bases and everyone else iterates on fine-tuning.

How they generate text.

One token at a time. At every step the model produces a probability distribution over ~150,000 possible next tokens, then samples one. The “output” is this loop run a few hundred times.

How deterministic

0.0 → always pick the top token (boring, predictable). 1.0 → sample faithfully from the distribution (creative, varied). Agents usually run at 0.0–0.3 for reliability.

top-p / top-k

Clip the long tail

Only consider the smallest set of tokens whose probabilities sum to p (e.g. 0.9). Kills the 0.001% nonsense tokens without flattening creativity.

max_tokens

Stop somewhere

Generation is a loop — you bound it. Also why “long” answers literally cost more: every token is another forward pass through the model (for Qwen3.6-35B-A3B, that's 3B active parameters per token, out of 35B loaded).

This is also why streaming works: the model is producing tokens one at a time anyway, so the UI can show them as they appear. And why structured output (JSON tool calls) is reliable — you constrain which tokens are even allowed next.

The context window.

Everything the model “knows” in the moment lives in its context window. Every new token can look back at every previous token. That mechanism — attention — is the whole magic trick.

the window

A scratch pad re-read every step

Prompt + conversation + tool outputs all live here. The model re-processes the whole thing to decide each next token. When the window fills up, older tokens get dropped or summarized.

attention

Every token sees every earlier token

For each new token, the model computes a weighted sum over all prior tokens — “how much does each one matter right now?” Strong matches dominate; irrelevant tokens fade out.

why it matters for agents

This is the short-term memory

When an agent “remembers” that it ran bjobs three steps ago, it's because that output is still sitting in the context window. Bigger window = longer chains of reasoning.

Attention, drawn

Each new token's attention over prior tokens. Thicker arc = stronger attention weight.

From LLM to Agent.

An LLM on its own is a brilliant text-completer — no more, no less. Three additions and a loop turn it into an agent. That's the whole rest of the talk.

Sidebar · Agentic Harnesses

The harness era.

We have passed the “ask a question in a chat window, copy the answer, paste it into your editor, run it, come back and apologize” stage. The useful model now lives inside your workflow — it sees your files, runs your tests, reads the errors, and tries again. That wrapper is called a harness.

2022 – 2023 · Chat era

Human is the courier

  1. Describe the bug in plain English.
  2. Paste the relevant file (hope it fits).
  3. Copy the answer back into your editor.
  4. Run it. It fails.
  5. Paste the error message back into chat.
  6. Repeat until tired.
→ harness
2024 – 2026 · Harness era

Agent is in the room

  1. Describe the bug once, in your editor or terminal.
  2. Agent reads the repo itself (grep, open, git blame).
  3. Agent runs the tests itself and reads the output.
  4. Agent edits files, re-runs, diffs the result.
  5. You review a single, coherent patch.
  6. No prompt-engineering ritual — the tools close the loop.
Harness = LLM  +  your tools  +  a tight feedback loop. The chat box was fine when the LLM had nothing but words. Once it can read files, run commands, and see what happened, you want it where the work is.

Why this is a big deal

Feedback compresses from minutes to seconds

In the chat era, the loop ran through a human — the most expensive, slowest element in the system. In the harness era the loop closes inside the machine. The model can try five variations of a fix in the time it took you to paste a stack trace. That’s the actual unlock; the model didn’t suddenly get smarter.

A failing test, fixed in place.

One prompt, seven tool calls, a real patch. No copy-paste, no context-ferrying. This is an abridged transcript from a harness session — the kind of interaction that used to take twenty minutes of editor/chat shuffling.

~/repo/scheduler  ·  claude-code · session #428
You pytest is failing on test_scheduler.py::test_requeue_on_preempt — can you figure out why and fix it?
Agent I’ll reproduce the failure first, then trace it.
Tool · shell $ pytest tests/test_scheduler.py::test_requeue_on_preempt -x
stdout FAILED — AssertionError: expected queue length 3, got 2 scheduler.py:142: when a job is preempted, it should be re-queued at the head
Tool · read scheduler.py  (lines 120–160)
Agent Found it — requeue() does queue.append(job), but the spec (and the test) expect the preempted job to go to the front. That’s a one-line fix: appendleft instead of append.
Tool · edit scheduler.py  — - queue.append(job)  /  + queue.appendleft(job)
Tool · shell $ pytest tests/test_scheduler.py -q

Notice what you didn’t do. You never pasted the file, you never pasted the traceback, you never pasted the fix back. The harness did the ferrying — the model just thought and acted.
